## 2026-05-17 업무 구조

### 데이터 접근 범위

- 모든 데이터는 몰(Mall)을 기준으로 분리 관리한다.
- 몰 운영자는 본인 몰 데이터만 접근 가능하다.
- 채널 운영자는 본인 채널 데이터만 접근 가능하다.
- 판매/구매회원은 본인 업체 데이터만 접근 가능하다.
- 다른 몰 데이터 접근은 시스템적으로 차단한다.

### 기본 운영 흐름

1. 수퍼관리자가 몰 기본정보를 최초 등록한다.
2. 수퍼관리자가 몰 운영자 정보를 등록하고, 해당 몰의 운영 권한을 부여한다.
3. 몰 운영자는 권한이 부여된 몰의 기본정보를 관리하고 채널을 생성한다.
4. 몰 운영자는 채널 담당자를 등록한 뒤 채널과 담당자를 매핑한다.
5. 채널 운영자는 본인에게 매핑된 채널만 관리한다.
6. 채널 운영자는 해당 채널의 판매업체, 구매업체, 판매회원, 구매회원, 상품, 주문, 출고, 정산 업무를 관리한다.

### 채널 기준

- 몰은 채널 타입 3가지까지만 채널을 생성 관리 할 수 있다.
- 채널 타입은 전국택배, 화물직배송, 공장수령 3가지이다.
- 하나의 몰 안에서는 동일한 채널 타입을 중복 생성할 수 없다.
- 채널 담당자는 `CHANNEL_ADMIN` 권한의 운영자만 지정할 수 있다.
- 이미 다른 채널에 배정된 채널 담당자는 운영자관리 화면에서 일반 운영자로 변경하지 못하도록 제한한다.

### 업체 기준

- 판매업체는 기본정보와 담당자 정보를 가진다.
- 판매업체는 원물, 가공, 완제품 중 하나 이상을 취급할 수 있다.
- 원물은 면세 기준, 가공/완제품은 과세 기준으로 처리한다.
- 구매업체는 기본정보와 담당자 정보를 가진다.
- 구매업체의 미트머니는 여신금액과 입출금 잔액을 합산하여 표시한다.
- 구매등급은 일반, 실버, 골드, 다이아몬드, VIP, VVIP 기준으로 운영한다.
- 구매등급별 할인율은 설정에서 별도로 관리하는 구조가 필요하다.
- 내부 금액 데이터는 여신, 입금, 출금, 사용, 정산 항목을 분리 관리해야 한다.

### 상품 기준

- 상품은 원물, 가공, 완제품 중 하나의 타입으로 등록한다.
- 상품 타입에 따라 과세/면세 기준과 할인, 정산, 주문 처리 기준이 달라질 수 있다.

### 미트머니 기준

- 미트머니는 구매 가능 금액 표시용 개념이다.
- 실제 내부 원장은 다음 항목으로 분리 관리한다.
  - 여신한도
  - 충전금
  - 사용금
  - 환불금
  - 정산대기금
  - 지급완료금
- 화면 표시용 미트머니는 내부 계산 결과를 합산하여 제공한다.

## AI 작업 원칙

- 모든 기능은 권한 범위를 기준으로 구현한다.
- 조회/수정/삭제는 반드시 몰 및 채널 범위 검증을 수행한다.
- 관리자 기능과 일반 회원 기능은 분리한다.
- 과세 기준은 상품 타입 기준으로 처리한다.
- 화면 표시용 금액과 실제 회계 원장은 분리한다.
- 동일 몰 내 채널 타입 중복 생성은 허용하지 않는다.
- 운영자 권한 변경 시 채널 매핑 여부를 반드시 검증한다.

- 몰 기본정보 관리
  - 몰 기본정보 조회/수정
  - 몰 기본정보 화면 안에서 담당자 목록 조회

  - 담당자 목록에는 채널 담당자(`CHANNEL_ADMIN`)가 노출되지 않도록 분리
  - 주소 검색은 다음 우편번호 레이어 방식으로 처리
- 채널관리
  - 채널 목록/등록/수정/삭제
  - 채널 목록에서 채널명 또는 담당자명을 클릭하면 상세/수정 팝업 표시
  - 채널 타입 중복 생성 제한
  - 채널 담당자 선택 기능
  - 채널 화면 안에서 신규 채널 담당자 계정 생성 및 즉시 지정
  - 신규 채널 담당자 입력 시 로그인 ID, 이름, 연락처, 이메일 길이와 형식 제한
  - 신규 채널 담당자 연락처는 입력 중 `010-1234-5678` 형식으로 자동 정리
  - 채널 담당자는 본인에게 배정된 채널 범위로 관리 권한 제한
- 운영자관리
  - 몰 운영자와 채널 담당자 목록 조회
  - 운영자 목록에서 이름 또는 로그인 ID를 클릭하면 상세/수정 팝업 표시

### 상품 관리 내용

- 상품유형 : 원물, 가공, 완제품 라디오버튼으로 표출
- 과세여부 : 원물일 경우 면세, 가공/완제품일 경우 과세로 설정
- 판매자명 : 몰 운영자는 판매자 목록에서 선택, 판매자는 본인
- 채널 : 판매자에 따른 채널
- 상품명 : 상품명 입력
- 축종구분 : 소, 돼지(코드에서 가져옴)
- 저장유형 : 냉장, 냉동(코드에서 가져옴)
- 처리유형 : 코드에서 가져옴
- 분리유형 : 코드에서 가져옴
- 단위/단량 구분 : 코드에서 가져옴
- 품목상태 : 판매중, 판매중지, 품절, 승인대기, 임시저장(코드에서 가져옴)
- 판매시작일, 판매종료일
- 공급가액 : 공급가액입력
- 판매가격 : 원물(면세)일 경우 공급가액, 가공/완제품(면세)일 경우 공급가액 + 부가세(판매가격 입력시 공급가액 자동계산)
- 전시순번 : 상품의 전시 순번(낮은 번호 먼저) 
  - 원물은 1000000, 완제품은 2000000, 가공은 3000000 번호대를 사용
  - 입력이 없으면 저장시 높은 번호의 100단위 UP(가령 원물의 높은번호가 1001122라면 1001200을 부여)
- 상품설명 : 네이버 웹에디터 
  - XSS 필터링
  - 이미지 업로드 제한
  - iframe 차단
  - script 제거
- MD코멘트 : 텍스트 입력
- 상품요약 : 텍스트 입력
- 배송비 : 택배일 경우 등록된 배송비
- 현재재고수량, 안전재고수량
- 상품번호 : 저장시 자동 채번
- 상품이미지 : 대표이미지 1개, 추가이미지 3개 등록 가능
- 신상품여부
- 이달의 상품여부
- 상품 숨김여부
- 추천가공 : 원물 선택시 가공서비스 선택팝업을 통한 여러개 선택 가능

## 2026-05-18 상품 관리 수정 반영 내용

- 상품 마스터 등록/수정 화면에서 상품유형을 원물/완제품/가공 라디오버튼으로 선택하도록 반영했다.
- 상품유형 선택 시 원물은 면세, 완제품/가공은 과세로 자동 설정되며 저장 시 서버에서도 동일 기준으로 보정한다.
- 몰 운영자/채널 운영자는 판매업체 목록에서 판매자를 선택하고, 판매자 권한은 본인 판매업체로 고정되도록 반영했다.
- 상품번호를 입력하지 않고 저장하면 상품유형별 번호대(원물 1000000, 완제품 2000000, 가공 3000000)를 기준으로 가장 높은 번호의 다음 100단위 번호를 자동 부여하도록 반영했다.
- 상품설명 저장 시 script, iframe, inline 이벤트, javascript: URL을 제거하는 1차 XSS 필터를 서버 저장 로직에 반영했다.
- 상품 목록에는 판매자명, 상품유형, 과세여부, 품목상태를 업무 용어로 표시하도록 정리했다.
- 공급가액/판매가격, 판매기간, 배송비, 안전재고, 대표/추가 이미지 URL, 신상품/이달의 상품/숨김여부, 추천가공 다중 선택은 `tb_product` 확장 컬럼과 `tb_product_image`, `tb_product_relation` 기반으로 저장하도록 반영했다.

## 2026-05-18 상품 관리 확장 수정 반영 내용

- `tb_product` 확장 컬럼 기준으로 공급가액, 부가세액, 판매가격, 판매시작일/종료일, 전시순번, MD코멘트, 상품요약, 배송비, 안전재고수량, 신상품/이달의상품/숨김여부를 상품 등록/수정 화면과 저장 로직에 연결했다.
- 판매가격은 원물(면세)은 공급가액과 동일하게, 완제품/가공(과세)은 공급가액 + 10% 부가세로 화면과 서버 저장 로직에서 자동 계산하도록 반영했다.
- 전시순번은 미입력 시 상품유형별 번호대(원물 1000000, 완제품 2000000, 가공 3000000)에서 가장 높은 번호의 다음 100단위 번호를 자동 부여하도록 반영했다.
- `tb_product_image` 테이블에 대표이미지 1개와 추가이미지 3개 URL을 저장하도록 반영했다.
- 추천가공은 원물 상품에서 가공서비스 상품을 선택해 `tb_product_relation`에 저장하도록 반영했다. 현재 로컬 DB의 `relation_kind` enum에는 `RECOMMENDED_PROCESS`가 없어 실제 저장값은 기존 enum인 `RAW_TO_PROCESS_SERVICE`와 `bundle_mode='OPTIONAL'`을 사용한다.
- 현재재고수량은 README의 추가 컬럼 SQL에 저장 컬럼이 없어 이번 구현 범위에서는 제외했다.
