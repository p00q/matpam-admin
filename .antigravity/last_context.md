### **수정 설계서: 데이터 정합성 검증 결과 미노출 핫픽스**

#### 1. 권장안 (Recommendation)

PRD에 명시된 대로, **JSP View Layer의 변수명만 수정**하여 문제를 해결합니다. 이는 Controller, Service, DAO 레이어의 변경을 전혀 유발하지 않아 가장 안전하고 빠른 해결책입니다.

**수정 대상 파일 및 내용:**

1.  **`webapp/WEB-INF/jsp/egovframework/validator/dataValidationResult.jsp`**
    -   JSTL `<c:forEach>` 루프 내에서 `egovMap`의 데이터를 참조하는 모든 EL(Expression Language) 변수명을 `camelCase`에서 `snake_case`로 변경합니다.
2.  **`webapp/WEB-INF/jsp/egovframework/validator/dataValidationList.jsp`**
    -   검색 폼 내 `pageIndex` hidden input의 `value`를 `1`로 명시적으로 설정합니다.

#### 2. 이유 (Reason)

*   **Side Effect 최소화**: 이 방식은 오직 화면 표시 로직만 수정합니다. 데이터 처리 로직, 비즈니스 로직, 데이터베이스 쿼리는 전혀 건드리지 않으므로, 새로운 기능적 결함이 발생할 가능성이 극히 낮습니다.
*   **신속성**: 수정 범위가 JSP 파일 2개로 한정되어 개발 및 테스트 시간이 매우 짧습니다. Critical 결함에 대한 가장 빠른 대응 방식입니다.
*   **명확성**: 문제의 원인(`egovMap` 키-값 불일치)과 해결책(JSP 변수명 일치)이 1:1로 대응하여, 누가 보더라도 수정 의도를 명확하게 파악할 수 있습니다.

#### 3. 대안 (Alternatives)

*   **대안 1: SQL Alias 변경**
    *   **내용**: `DataValidation_SQL.xml`의 SELECT 쿼리에서 `SELECT ORDER_ID AS orderId, ...` 와 같이 Alias를 `camelCase`로 변경합니다.
    *   **기각 사유**: 데이터베이스의 표현(snake_case)을 특정 화면의 요구(camelCase)에 맞추는 것은 좋은 설계가 아닙니다. 만약 다른 서비스에서 해당 쿼리를 `egovMap`으로 재사용할 경우, 예상치 못한 키 값으로 인해 또 다른 문제를 유발할 수 있습니다.

*   **대안 2: MyBatis `mapUnderscoreToCamelCase` 설정 활성화**
    *   **내용**: MyBatis 설정 파일(`sql-map-config.xml`)에 `<setting name="mapUnderscoreToCamelCase" value="true"/>`를 추가합니다.
    *   **기각 사유**: 이는 프로젝트 전역에 영향을 미치는 매우 위험한 변경입니다. 현재 `snake_case` 키를 정상적으로 사용하고 있는 수많은 다른 기능들에서 연쇄적인 장애를 일으킬 수 있습니다. 핫픽스에서 절대 선택해서는 안 될 방식이며, 도입하려면 전수 테스트가 동반된 별도의 대규모 프로젝트로 진행해야 합니다.

#### 4. 트레이드오프 (Trade-off)

*   **선택**: 권장안을 채택하면, 해당 JSP 페이지는 `snake_case` 변수명을 사용하게 되어 프로젝트 내 다른 JSP 파일들과의 일관성이 다소 저하될 수 있습니다.
*   **결론**: 하지만 이는 사소한 코드 스타일의 불일치일 뿐, 기능적 안정성을 확보하는 것이 훨씬 중요합니다. PM님께서 제안하신 대로, 장기적으로 MyBatis 설정을 표준화하는 과제를 백로그에 추가하여 근본적인 해결을 도모하는 것이 올바른 순서입니다.

#### 5. 리스크 및 완화 전략 (Risks & Mitigation)

*   **리스크 1: 개발자의 `mapUnderscoreToCamelCase` 설정 임의 확인 누락**
    *   **내용**: 만약 개발 환경에 해당 설정이 `true`로 되어 있다면, 권장안대로 수정 시 오히려 화면이 깨지게 됩니다.
    *   **완화 전략**: 개발 착수 전, 담당 개발자는 반드시 로컬 및 개발 서버의 MyBatis 설정 파일에 `mapUnderscoreToCamelCase` 옵션이 없는지(또는 `false`인지) **스크린샷으로 확인하여 공유**하는 것을 의무화합니다.

*   **리스크 2: 회귀 결함 발생**
    *   **내용**: 변수명 수정 과정에서 오타가 발생하거나, 기존에 정상 동작하던 Ajax 연동, 페이징 순번 계산 로직에 영향을 줄 수 있습니다.
    *   **완화 전략**: 개발 완료 후, 개발자는 PRD의 **Acceptance Criteria(AC-1 ~ AC-4) 전체 항목**을 직접 실행하고 결과를 증빙으로 첨부해야 합니다. QA팀은 수정된 부분뿐만 아니라, 해당 기능의 전체 시나리오를 다시 한번 검증해야 합니다.
